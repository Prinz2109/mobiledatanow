How To

This document describes at a high level how to configure Mobile Data Now to deliver data to mobile devices.
This document acts primarily as an index to other help documents that describe in detail each of the screens and processes used to administer Mobile Data Now.

Before you can use Mobile Data Now you must first install and licence MDN as described in the Installation and Configuration guide and start the MDN Server.

Also see MDN Administrator and Definitions and Terminology for more information on using the MDN Administrator and the terms and terminology used in this documentation.


Using MDN Overview

To deliver data to mobile users you must perform the following sequence of steps using the MDN Administrator:

1. Start the Administrator and Login.
2. Import Data Store Definitions.
3. Build Data Views and Queries.
4. Create Groups, Users and Permissions.
5. Build Presentation Menus.


Mirroring and Data Transfers

To make use of the optional Mirror Data Store and Data Transfer features you must:

1. Create Mirror Data Stores.
2. Mirror External Data Stores.
3. Create Data Transfers.
4. Schedule Data Transfers (optional).

Start the Administrator and Login

To start the MDN Administrator choose Mdn Administrator from the Start|Programs|Mobile Data Now menu under Windows, or type mdnadmin from a command shell window under Linux/Solaris.
The MDN Administrator will display an error message and exit if the MDN Server is not running or if MDN has not been licensed.
Enter a username and password and log in.


Import Data Store definitions

Use the Data Store Wizard to create a new Data Store to access your database and import the table definitions of the tables you wish to present with MDN.


Build Data Views and Queries

Use the View Wizard to create one or more views of your Data Stores.

Use the Query Wizard to create one or more queries to select data from a view for presentation to your users.


Create Groups, Users and Permissions

Use the Groups screen to create the groups to which your users will belong. Each different group can be configured with a different set of menus using the Presentation Builder.

Use the Users screen to create a user account for each user and to assign the users to groups.


Build Presentation Menus

Use the Presentation Builder to build a menu structure for each group.


Mirroring and Data Transfers

Mirroring is optional feature of MDN that allows you to create a mirror copy of your source data store. MDN will then access the mirror instead of the source data store. This can be useful to reduce load or protect the source data store.
You can use automatically scheduled or manual data transfers to populate the mirror with data from the source data store.


Create Mirror Data Stores

Use the Mirror Data Stores screen to create a Mirror Data Store.


Mirror External Data Stores

In the Data Stores screen select a Data Store, press the Properties button to display the JDBC Data Store properties screen. Check the Mirrored Data Store? check box then select the desired Mirror Data Store to activate mirroring for this Data Store.


Create Data Transfers

Use the Data Transfers screen to create a Data Transfer that will populate the Mirror Data Store with the desired data from the source (mirrored) Data Store. Press the Execute button to execute the transfer and copy the data.


Schedule Data Transfers (optional)

If you want the Data Transfer to be run periodically use the Schedule screen to schedule a Data Transfer for future execution.


Tips

Keep it small. Mobile devices and WAP phones usually have a very small screen. When defining views use as few fields, especially naming fields, as possible and keep the field display names short.

Delete extraneous fields from your data stores. When the data store wizard imports a table it gets all fields. If you do not need to use a field in a view, and it is not a key fields for the table then it is safe to delete it from the data store. This will not affect the source table.

Assign keys for the tables in your data stores. Update and delete operations in MDN may not work correctly if MDN cannot uniquely identify the correct record.

Show keys in the views or use auto key generation. When a new record is added in MDN it must have valid keys which are unique. The user needs to be able to enter values for the keys and any other fields that are required by your database, otherwise the database will generate an error when inserting or updating the record.
Alternatively the database could generate the keys using a trigger or auto-number field. In this case the key field(s) need to be in the data store so MDN can uniquely identify the record, but need not be shown in the view.

Use mirrors. If the data you are presenting with MDN is read-only, the source database is heavily loaded or for security reasons you wish to reduce MDN's access to the source database, then you can use mirror data stores and have MDN work from the mirror copy. It is usually best to use a separate mirror for each data source because sharing a mirror between data sources may degrade performance and will cause an error if the different data sources have tables with the same name.


Testing

After you have defined you data stores, views and menus it is necessary to test your MDN system. Most functions will work correctly, but it is possible that database errors will occur if required fields or keys are omitted from the views or if relationships have not been specified correctly.
Using the MDN Administrator select each query in each view and execute it by pressing the Execute button.
Use a WAP browser or web browser to navigate through all the menu options in your MDN system to ensure that they work correctly.


Troubleshooting

Errors when inserting or updating records may be due to a key or constraint error in the underlying database. This can be caused by not correctly specifying the keys for the data store entities, not including all required fields and keys in the view or the user entering a key value which is already in use.
It is preferable in many cases for the database to generate the keys using triggers or auto-number fields rather than requiring the user to enter the key data.

An error when inserting records during a data transfer may be caused if two data stores use the same mirror and they have a table name in common. Since a mirror can not have two tables with the same name one definition will be overwritten, leading to errors when transferring or viewing the data in the data store.
To resolve this problem ensure that data stores that share a mirror do not have any tables with the same name. It is preferable to use a separate mirror for each mirrored data store.

Some databases treat tables, views and queries in a similar manner. Because of this it may be possible to use the data store wizard import a "table" that is in fact a query. Is may cause problems if the database does not allow some operations, such as insert or update, to be performed on a query. There will also be problems if the query requires run time parameters to execute. In this case you will see an SQL exception with the message Too few Parameters.
